# Mesh VPN

Shuttle 的 Mesh VPN 为每个客户端分配一个虚拟 IP（VIP），允许客户端之间直接点对点通信——当无法建立直连时，服务器作为中继回退。

---

## 架构

```
          ┌──────────┐
          │  服务端  │  （中继 + 信令）
          └────┬─────┘
         ┌─────┴─────┐
    ┌────┴────┐ ┌────┴────┐
    │client-a │ │client-b │
    │10.7.0.3 │ │10.7.0.2 │
    └─────────┘ └─────────┘
         ↑___直连 P2P___↑
```

**星型中继：** 所有客户端连接到服务端，服务端可在任意两个客户端之间转发数据包。

**P2P 升级：** 注册后，客户端通过 NAT 穿透尝试建立直连路径。成功后流量直接在端对端间传输，不再经过服务端。

---

## NAT 穿透流程

Shuttle 按顺序尝试以下机制，成功即停止：

1. **mDNS** — 通过组播 DNS 发现同局域网内的对端。
2. **STUN** — 获取每个对端的公网端点。
3. **UPnP / NAT-PMP** — 向网关申请端口映射。
4. **打洞（Hole Punching）** — 双端同时发送 UDP 包以建立 NAT 映射。
5. **TURN 中继** — 所有直连路径均失败时，回退到服务端中继。

## 配置

### 服务端

```yaml
mesh:
  enabled: true
  cidr: 10.7.0.0/24       # VIP 地址池；服务端占用第一个地址
  mtu: 1420
```

### 客户端

```yaml
mesh:
  enabled: true
  vip: 10.7.0.3            # 请求的虚拟 IP（省略则由服务端自动分配）
  stun_servers:
    - stun:stun.l.google.com:19302
    - stun:stun1.l.google.com:19302
```

### 字段说明

**服务端字段：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | `false` | 启用 Mesh VPN |
| `cidr` | CIDR | 必填 | VIP 地址池 |
| `mtu` | int | `1420` | 虚拟网卡 MTU |

**客户端字段：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | `false` | 启用 Mesh VPN |
| `vip` | string | 自动 | 请求的虚拟 IP |
| `stun_servers` | list | 内置 | STUN 服务器 URI 列表 |

---

## 虚拟 IP 地址

连接成功后，每个客户端从服务端的 `cidr` 地址池中获得一个 VIP。对端可直接通过 VIP 访问，无需 DNS。服务端本身持有地址池中的第一个地址（如 `10.7.0.1`）。

三节点示例：

| 节点 | VIP |
|------|-----|
| 服务端 | 10.7.0.1 |
| client-a | 10.7.0.3 |
| client-b | 10.7.0.2 |

从 client-a 执行 `ping 10.7.0.2` 即可访问 client-b，走最优路径（直连 P2P 或中继）。
